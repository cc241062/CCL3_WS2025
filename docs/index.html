Oops! I can cook

A learning app that helps beginner cooks grow confident in the kitchen through step-by-step recipes, personal creations, and fun monthly quests.

Overview

Short description: Oops! I can cook is a learning app for beginner chefs. Users can create and edit their own profiles, learn pre-made recipes in a monthly “Quest mode”, add their own recipes, and search for them.

Team Members

  • Antonia Hruschka

App Concept

Use Case

Oops! I can cook is designed to teach beginners how to cook from scratch, build confidence in the kitchen, and stay motivated to prepare home-cooked meals instead of relying on takeout — all through step-by-step lessons, personal recipe creation, and fun monthly cooking quests.

Target User

Oops! I can cook is for inexperienced home cooks who want to start eating healthier and learn to cook for themselves.

User Flow / Hi-Fi Prototype

Users can:

  • Create and edit a personal profile.
  • Browse pre-made recipes and follow them step by step.
  • Enter “Quest mode” to complete a themed monthly cooking challenge.
  • Add their own recipes and search them later.

Heuristic Evaluation

The evaluation suggests that the app’s core structure is sound, but several small to medium usability problems accumulate and could noticeably slow down or confuse users—especially beginners, who are the target audience. Navigation clarity, consistency, and visibility of system status are the most prominent themes across the identified issues.

Addressing the high-severity items, such as navigation clarity, missing UI elements, and unclear dropdown behavior, will significantly improve the overall user experience. Enhancing the Quest screen’s visual clarity and motivational cues will further support the app’s goal of keeping users engaged and encouraging repeated use.

Hypotheses & Testable Questions

Main Hypothesis

H1: Visual cues in the interface significantly impact how users interact with the app, influencing navigation efficiency, content recognition, and overall enjoyment.

Testable Questions

Navigation

Do first-time users navigate to main features more successfully with or without onboarding instructions?

  • Independent Variable (IV): Presence of onboarding instructions
  • Dependent Variable (DV): Time needed to reach each main feature

Visual Design – Recipe Cards

Does using colored recipe cards improve users’ ability to distinguish recipes compared to neutral-colored cards?

  • Independent Variable (IV): Recipe card color (colored vs. neutral)
  • Dependent Variable (DV): Users’ ability to distinguish recipes (measured by accuracy and error rate)

Gamification – Quest Screen

Does a gamified quest screen design increase users’ perceived enjoyment when using the feature?

  • Independent Variable (IV): Presence of gamified elements on the quest screen
  • Dependent Variable (DV): Users’ perceived enjoyment

Data to Be Collected

Qualitative Data

  • Users’ subjective perceptions of visual design and navigation cues.
  • Identification of pain points, confusion, and frustrations related to the interface.
  • User expectations and emotional responses during task completion.

Quantitative Data

  • Task completion times for navigation and feature discovery.
  • Error rates during tasks (wrong clicks, backtracking, failed attempts).
  • Perceived usability scores measured through standardized questionnaires (e.g., SUS and SEQ).

This data will be used to compare different interface designs and determine which visual cues lead to more efficient, understandable, and enjoyable user interactions.

Methods & Data Collection

Study Design

  • Between-subjects design to compare different visual and interaction conditions.

Evaluation Methods

  • Moderated usability testing with a think-aloud protocol.
  • Direct observation of user behavior during task completion.
  • Screen recording or video recording when screen capture is not available.
  • Short interviews to explore user expectations and frustrations.

Questionnaires & Tools

  • SEQ (Single Ease Question): Administered after each task to measure perceived task difficulty.
  • SUS (System Usability Scale): Administered after all tasks to assess overall usability.
